Workflows & Database Operations

Every user action in an airline system triggers a sequence of reads and writes across multiple tables. These flowcharts trace the exact path each operation takes through the database — from the triggering event to the final state change.

A — Flight Booking Workflow

A passenger searches for a flight, selects a seat, and confirms a booking. Seven tables are touched in sequence. A hold prevents double-booking during checkout.

B — Check-in & Boarding Workflow

At the airport or online, the passenger checks in, receives a boarding pass, and is tracked at the gate. The checkin table holds one record per booking segment (enforced by UNIQUE on booking_id).

C — Payment & Refund Workflow

Payment is captured at the PNR level — one aggregated charge per order. Cancellations insert a new row into the refund table without altering the original payment record.

D — Crew Assignment Workflow

Operations assigns crew to a flight via the crew_assignment junction table, which resolves the M:N relationship between crew and flight. A scheduling conflict check runs before every insert.

Index Strategy

An index is a sorted auxiliary data structure that lets the database jump directly to matching rows instead of scanning every row in the table. Think of it as the index at the back of a textbook — instead of reading every page, you go straight to the reference.

Without an index

Full table scan — the engine reads every row from top to bottom. On a table with 10 million flights, a date-range query reads all 10 million rows even if only 200 match.

With a B-Tree index

The engine traverses a balanced binary tree in O(log n) steps to the first matching row, then reads only the matching range — typically a few hundred rows regardless of table size.

Trade-off

Every index speeds up reads but adds overhead to writes — on every INSERT, UPDATE, or DELETE, all relevant indexes must be updated too. This is why we index only the columns that serve the six most frequent queries, not every column.

The 6 custom performance indexes
# Index Name Table & Columns Type Purpose & Query it serves
01 idx_flight_route_date flight
route_id, scheduled_departure		 B-Tree Composite Serves the most frequent query: flights between two airports on a given date. route_id is the leftmost key (equality match), then scheduled_departure is range-scanned. Column order is intentional — reversing it would make the index useless for airport-pair searches.
02 idx_seat_inventory_lookup seat
flight_id, availability_status, fare_class_id, seat_number		 B-Tree Covering Serves available seats by fare class for a selected flight. All four columns are in the index, so the query is answered entirely from the index B-Tree — no row lookup needed. This is a covering index. Eliminates heap fetches during high-traffic checkout.
03 idx_checkin_boarding_alert checkin
boarding_status, booking_id		 B-Tree Full Serves passengers checked in but not yet boarded. MySQL does not support partial (filtered) indexes — this is a full B-Tree. The application adds WHERE boarding_status = 'checked_in'; the index prefix still enables a fast selective scan. In PostgreSQL this would be a filtered partial index.
04 idx_pnr_code pnr_group
pnr_code		 B-Tree UNIQUE Serves exact-match PNR lookups. MySQL InnoDB does not support explicit hash indexes — CREATE INDEX USING HASH is silently ignored and a B-Tree is created. InnoDB's Adaptive Hash Index (AHI) builds in memory automatically on hot pages, giving near-O(1) performance for repeated PNR lookups. UNIQUE also enforces the business rule that PNR codes are globally non-repeating.
05 idx_flight_delay_tracking delay_record
delay_minutes, flight_id		 B-Tree Composite Serves flights delayed beyond 2 hours. Application filters WHERE delay_minutes > 120; the index enables an efficient range scan from 120 upward. flight_id as suffix allows the dashboard to retrieve affected flight IDs directly from the index without a row lookup.
06 idx_crew_assignment_schedule crew_assignment
crew_id, assignment_datetime		 B-Tree Composite Serves scheduling conflict detection — a crew member cannot be on two flights in the same duty window. crew_id equality-matched first, then assignment_datetime range-scanned. Hit on every new crew assignment write — the most critical index on the write path.
04

High-Level Architecture

Strict BCNF normalisation is ideal for write integrity but creates deep multi-table join overhead on reads. The architecture solves this by separating the write path (MySQL — source of truth) from the read path (Redis / Elasticsearch — pre-joined cache), connected by an asynchronous event pipeline.

GDS Integration

Amadeus, Sabre, and Travelport route bookings through a GDS Adapter that normalises EDIFACT / NDC payloads into the internal schema. booking.source_channel tracks origin; booking.external_booking_ref stores the GDS cross-reference PNR.

CQRS Pattern

Command Query Responsibility Segregation. All writes (commands) go to the normalised MySQL instance. All reads (queries) are served from Redis or Elasticsearch — pre-joined documents, zero joins at read time. The two paths never share a bottleneck.

Event Pipeline

On every successful MySQL write, an event (e.g. BookingConfirmedEvent) is fired onto a message bus (RabbitMQ / Kafka). A consumer joins the relevant tables once and pushes a pre-built JSON document to Redis. The user's API response is not blocked by this step.
